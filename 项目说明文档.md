# 数字资产保险箱  系统开发计划文档

## 一、项目定位

本项目是一套面向个人用户的高安全私密信息管理系统，核心功能包括：

1. 存储账号密码
2. 存储股票账户、银行账户、邮箱账户、服务器账户等敏感信息
3. 私密相册、私密视频、私密文件、表格、合同、证件资料上传
4. 支持小程序端、PC Web 端登录
5. 未来支持 APP 端
6. 支持设置安全联系人
7. 支持设置数字遗产交接规则
8. 用户长期失联后，系统自动触发确认流程
9. 满足条件后，安全联系人通过验证后获得访问权限
10. 所有用户内容必须加密存储，开发者、管理员、服务器均无法查看明文内容
11.APP支持多语言切换使用。适用于不同国家使用

本产品不是普通网盘，也不是普通密码本，而是“零知识加密保险箱 + 意外身故/失联后的数字遗产交接系统”。

---

## 二、核心安全原则

### 1. 零知识加密原则

用户的所有私密内容必须在客户端加密后再上传服务器。

服务器只保存：

* 加密后的内容
* 加密后的文件
* 加密后的密钥包
* 用户基础账号信息
* 交接规则
* 安全联系人信息
* 登录状态与审计日志

服务器不得保存用户主密码明文，也不得保存可直接解密用户数据的密钥。

### 2. 开发者不可见原则

后台管理员、数据库管理员、服务器运维人员，即使拿到数据库和文件存储，也只能看到密文，不能看到用户的账号密码、图片、视频、文件内容。

### 3. 用户主密码不可找回原则

由于系统采用零知识加密，用户主密码不能被平台找回。

如果用户忘记主密码，只能通过：

* 用户提前设置的恢复密钥
* 多因子恢复机制
* 安全联系人联合恢复
* 预先设置的密钥托管方案

否则无法解密历史数据。

### 4. 多重验证原则

普通登录、安全联系人接管、账户恢复、导出数据、修改继承规则等高风险操作，必须强制进行二次验证。

## 三、推荐技术架构

### 前端

1. 微信小程序端
2. PC Web 端
3. 未来 APP 端

推荐技术：

* 小程序：uni-app 
* PC Web：Vue 3 + TypeScript + Vite
* APP：uni-app 

### 后端

推荐技术：

* Node.js + NestJS
* MySQL
* 对象存储：腾讯云 COS / 阿里云 OSS / S3
* 定时任务：NestJS Schedule (Cron)
* 邮件服务：SendCloud / 阿里云邮件 / 腾讯云邮件
* 短信服务：腾讯云短信 / 阿里云短信
* 审计日志：独立日志表 + 不可篡改日志策略

### 加密相关

前端使用 WebCrypto API 或成熟加密库：

* AES-256-GCM：加密用户数据和文件
* Argon2id：从用户主密码派生密钥
* HKDF：派生不同用途的子密钥
* X25519 / RSA-OAEP：用于安全联系人密钥加密
* Ed25519 / ECDSA：用于签名和验证关键操作
* HMAC：用于完整性校验

### 文件加密流程

1. 用户选择文件
2. 前端生成随机文件密钥 file_key
3. 使用 file_key 对文件进行 AES-256-GCM 加密
4. 上传加密文件到对象存储

6. 后端保存 encrypted_file_key
7. 后端永远无法解密原始文件

## 5. 数字遗产交接模块

### 交接触发条件

用户可以设置：

* 1 年未登录
* 2 年未登录
* 3 年未登录
* 自定义年限
* 指定日期后触发
* 手动触发
* 法律文件触发
* 多安全联系人联合申请触发

### 推荐默认规则

如果用户 3 年未登录：

第一年：每 3 个月发送一次提醒
第二年：每 2 个月发送一次提醒
第三年：每 1 个月发送一次提醒
满 3 年后：进入安全联系人验证期
验证期 1 年内仍无用户回应：允许安全联系人申请接管

### 系统确认方式

系统自动发送：

* 邮件
* 短信
* 微信服务通知
* 小程序订阅消息
* PC 端登录提示

### 用户回应方式

用户可以点击：

* 我还在使用，取消交接流程
* 暂停交接 1 年
* 修改交接规则
* 立即允许安全联系人接管
* 关闭数字遗产功能

## 6. 安全联系人接管流程

### 接管流程

1. 安全联系人收到通知
2. 进入安全联系人验证页面
3. 输入手机号/邮箱验证码
4. 完成身份问题验证
5. 回答用户预设问题
6. 上传身份材料，选做
7. 系统进入冷静期
8. 冷静期继续通知原用户
9. 原用户无回应
10. 系统释放加密密钥访问权限
11. 安全联系人可登录查看授权范围内内容

### 验证问题设计

用户可以设置多个问题，例如：

* 我们第一次见面的城市？
* 我的出生地？
* 我的第一只宠物叫什么？
* 我的备用邮箱是什么？
* 我最信任的人是谁？

注意：问题答案本身也要加密存储，服务器不能看到明文答案。

## 7. 密钥设计方案

### 核心密钥

每个用户创建账户时生成：

* master_key：由用户主密码派生，不上传服务器
* vault_key：用于加密用户保险箱数据
* file_key：每个文件单独生成
* recovery_key：用户恢复密钥
* contact_access_key：安全联系人接管密钥

### 加密关系

用户主密码
→ Argon2id 派生 master_key
→ 解密 encrypted_vault_key
→ vault_key 解密用户数据
→ vault_key 解密 file_key
→ file_key 解密文件

### 安全联系人密钥

安全联系人不能直接拿到 vault_key。

系统应保存：

* encrypted_vault_key_for_contact
* contact_verify_policy
* contact_unlock_status
* unlock_available_at

只有在满足交接条件后，安全联系人完成验证，前端才能解密对应密钥包。
